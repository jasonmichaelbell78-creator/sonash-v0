# Recall Reference

Query mode specifications, output formatting rules, follow-up action contracts,
auto-rebuild procedure, and example queries for the recall skill.

---

## 1. Query Modes

The `/recall` skill parses input into one or more query modes. Modes can be
combined freely. The underlying script is `scripts/cas/recall.js`, which outputs
JSON. The skill transforms that JSON into human-readable presentation.

### 1.1 Free-Text Search

**Trigger:** Any non-flag argument(s).

**Mechanism:** FTS5 full-text search against the `search_extractions` virtual
table (columns: `candidate`, `notes`, `tags`). Uses SQLite FTS5 MATCH syntax
with the `porter unicode61` tokenizer, which provides stemming (e.g.,
"extracting" matches "extraction") and Unicode normalization.

**SQL (generated by recall.js):**

```sql
SELECT e.candidate, e.type, e.source, e.source_type, e.novelty,
       e.effort, e.relevance, e.decision, e.decision_date, e.notes, e.tags
FROM extractions e
WHERE e.rowid IN (
  SELECT rowid FROM search_extractions
  WHERE search_extractions MATCH ?
)
ORDER BY e.id DESC
LIMIT ?
```

**FTS5 query syntax supported:**

| Syntax        | Example                  | Meaning                    |
| ------------- | ------------------------ | -------------------------- |
| Simple terms  | `architecture patterns`  | AND of stemmed terms       |
| Quoted phrase | `"plugin architecture"`  | Exact phrase match         |
| OR            | `react OR vue`           | Either term                |
| NOT           | `architecture NOT react` | Exclude term               |
| Prefix        | `arch*`                  | Prefix match               |
| Column filter | `candidate:routing`      | Search specific FTS column |

**Notes:**

- The porter stemmer means "routing" matches "route", "routed", "router".
- FTS5 ranks results by BM25 internally, but `recall.js` currently orders by
  `e.id DESC` (insertion order). Sort overrides are applied via `--sort`.

### 1.2 Tag Filter

**Trigger:** `--tag=<tagname>`

**Mechanism:** Joins through the `extraction_tags` and `tags` junction tables.
Tags are stored lowercase without `#` prefix.

**SQL (generated by recall.js):**

```sql
SELECT ... FROM extractions e
WHERE e.id IN (
  SELECT et.extraction_id FROM extraction_tags et
  JOIN tags t ON et.tag_id = t.id
  WHERE t.name = ?
)
```

**Tag normalization:** The `--tag` value is lowercased and stripped of leading
`#` before querying. So `--tag=#Architecture`, `--tag=architecture`, and
`--tag=ARCHITECTURE` all produce the same query.

**Combining with free-text:** When both free-text and `--tag` are provided, both
conditions are AND-ed:

```sql
WHERE e.rowid IN (SELECT rowid FROM search_extractions WHERE ... MATCH ?)
  AND e.id IN (SELECT et.extraction_id FROM extraction_tags et ...)
```

### 1.3 Type Filter

**Trigger:** `--type=<source_type>`

**Mechanism:** Direct column filter on `extractions.source_type` or
`sources.source_type`.

**Valid values:** `repo`, `website`, `document`, `media`

**SQL (extraction query):**

```sql
WHERE e.source_type = ?
```

**SQL (source query, when combined with `--target=sources`):**

```sql
WHERE s.source_type = ?
```

### 1.4 Sort Modes

**Trigger:** `--sort=<mode>`

**Available modes:**

| Mode      | SQL ORDER BY clause                                                                     | Description              |
| --------- | --------------------------------------------------------------------------------------- | ------------------------ |
| `recent`  | `ORDER BY e.decision_date DESC`                                                         | Newest extractions first |
| `novelty` | `ORDER BY CASE e.novelty WHEN 'high' THEN 1 WHEN 'medium' THEN 2 WHEN 'low' THEN 3 END` | High novelty first       |

**Default sort (no --sort flag):** `ORDER BY e.id DESC` (insertion order, most
recently added first).

**For source queries (`--target=sources`):** The sort mode `recent` maps to
`ORDER BY s.analyzed_at DESC`. The `novelty` sort is not available for source
queries (sources do not have a novelty field).

### 1.5 Source Filter

**Trigger:** `--source=<slug-or-partial>`

**Mechanism:** LIKE match against `extractions.source`.

**SQL:**

```sql
WHERE e.source LIKE ?
-- Parameter: %<source-value>%
```

This allows partial matching. `--source=next` matches sources containing "next"
anywhere in the source string (e.g., `vercel/next.js`, `nextcloud/server`).

### 1.6 Source Listing Mode

**Trigger:** `--target=sources`

**Mechanism:** Queries the `sources` table instead of `extractions`.

**SQL:**

```sql
SELECT s.title, s.source_type, s.source, s.slug, s.depth,
       s.quality_band, s.quality_score, s.personal_fit_band,
       s.personal_fit_score, s.classification, s.summary, s.tags
FROM sources s
[WHERE conditions]
ORDER BY s.analyzed_at DESC
LIMIT ?
```

**Compatible filters:** `--tag`, `--type`, and free-text (via `search_sources`
FTS5 table). `--source` and `--sort=novelty` are not applicable to source
queries.

### 1.7 Stats Mode

**Trigger:** `--stats`

**Mechanism:** Runs aggregate queries and returns a summary object.

**SQL queries executed:**

```sql
-- Total counts
SELECT COUNT(*) as count FROM sources;
SELECT COUNT(*) as count FROM extractions;
SELECT COUNT(*) as count FROM tags;

-- Breakdown by source type
SELECT source_type, COUNT(*) as count
FROM sources GROUP BY source_type ORDER BY count DESC;

-- Breakdown by novelty
SELECT novelty, COUNT(*) as count
FROM extractions GROUP BY novelty ORDER BY count DESC;

-- Top 15 tags
SELECT t.name, COUNT(*) as count
FROM extraction_tags et JOIN tags t ON et.tag_id = t.id
GROUP BY t.name ORDER BY count DESC LIMIT 15;
```

**Output JSON structure:**

```json
{
  "summary": {
    "sources": 29,
    "extractions": 89,
    "unique_tags": 47
  },
  "by_source_type": [
    { "source_type": "repo", "count": 24 },
    { "source_type": "website", "count": 5 }
  ],
  "by_novelty": [
    { "novelty": "high", "count": 34 },
    { "novelty": "medium", "count": 41 },
    { "novelty": "low", "count": 14 }
  ],
  "top_tags": [
    { "name": "architecture", "count": 12 },
    { "name": "extraction", "count": 9 }
  ]
}
```

### 1.8 Result Limit

**Trigger:** `--limit=<number>`

**Default:** 20

**Applied to:** Both extraction and source queries as the SQL `LIMIT` clause.
Stats mode ignores this flag.

## 2. Output Format

The `recall.js` script outputs raw JSON to stdout and a result count to stderr.
The `/recall` skill transforms this JSON into human-readable presentation for
the user.

### 2.1 Extraction Results (default)

**JSON fields per result:**

```json
{
  "candidate": "Auto-routing via type detection",
  "type": "pattern",
  "source": "unstructured-io/unstructured",
  "source_type": "repo",
  "novelty": "high",
  "effort": "E2",
  "relevance": "high",
  "decision": "defer",
  "decision_date": "2026-04-05",
  "notes": "partition() auto-detects file type via libmagic...",
  "tags": "[\"extraction\",\"architecture\",\"pattern\"]"
}
```

**Presentation format:**

```
Found 8 results for "architecture":

1. Auto-routing via type detection [pattern]
   Source: unstructured-io/unstructured (repo) | Novelty: high | Effort: E2
   "partition() auto-detects file type via libmagic, routes to format-specific
   partitioner."

2. Per-site extractor plugin architecture [pattern]
   Source: iawia002/lux (repo) | Novelty: high | Effort: E2
   "44-site video downloader with per-site extractor plugins."

3. ...
```

**Formatting rules:**

- **Line 1:** Result number, candidate name in bold, type in brackets
- **Line 2:** Source name with source type in parens, novelty, effort
- **Line 3:** Notes as a quoted excerpt (truncate to ~120 chars if longer)
- **Separator:** Blank line between results
- **Header:** "Found N results for [query/filter description]"
- **Tags field:** Parse the JSON string array before display; show as
  comma-separated if presenting inline

### 2.2 Source Results (`--target=sources`)

**JSON fields per result:**

```json
{
  "title": "Unstructured",
  "source_type": "repo",
  "source": "unstructured-io/unstructured",
  "slug": "unstructured",
  "depth": "standard",
  "quality_band": "Excellent",
  "quality_score": 82,
  "personal_fit_band": "Excellent",
  "personal_fit_score": 85,
  "classification": "active-sprint",
  "summary": "Open-source toolkit for ingesting and pre-processing...",
  "tags": "[\"extraction\",\"python\",\"etl\"]"
}
```

**Presentation format:**

```
29 analyzed sources:

  repo (24): unstructured, docling, reader, crawl4ai, firecrawl, ...
  website (5): docs-composio-dev, karpathy-gist, ...
  document (0)
  media (0)
```

**Formatting rules:**

- **Group by source type.** Show type name with count, then comma-separated
  slugs.
- **Sort within groups** by `quality_score` descending.
- **Truncate slug lists** at 8 items with "..." if more exist.
- **Total count** as header: "N analyzed sources:"

**Detailed source listing (when result count is 5 or fewer):**

When the result set is small enough, show full detail per source:

```
3 sources matching --type=website:

1. docs-composio-dev | Healthy (72) | active-sprint
   "Composio documentation site covering 200+ tool integrations..."
   Tags: website, documentation, integrations

2. karpathy-gist | Excellent (88) | active-sprint
   "Andrej Karpathy's educational gist on training neural nets..."
   Tags: website, ml, education
```

### 2.3 Stats Results (`--stats`)

**Presentation format:**

```
Content Analysis Index Statistics

  Sources:     29
  Extractions: 89
  Unique tags: 47

  By source type:
    repo       24
    website     5
    document    0
    media       0

  By novelty:
    high       34  (38%)
    medium     41  (46%)
    low        14  (16%)

  Top tags:
    architecture (12), extraction (9), python (7), react (6),
    pattern (5), ...
```

**Formatting rules:**

- **Aligned columns** for readability.
- **Percentages** on novelty breakdown (computed from total extractions).
- **Top tags** as inline comma-separated with counts in parens.
- **Zero counts shown** for source types with no entries (helps user see what is
  not yet analyzed).

### 2.4 Empty Results

When no results match the query:

```
No results found for "quantum computing".

Suggestions:
- Broaden your search terms
- Check available tags: /recall --stats
- List all sources: /recall --target=sources
- Verify the index is current: the index was last rebuilt on [date]
```

## 3. Follow-Up Actions

After presenting results, the skill offers contextual follow-up options. The
available options depend on the query mode and result set.

### 3.1 After Extraction Results

| Option | Label                | Action                                                              |
| ------ | -------------------- | ------------------------------------------------------------------- |
| 1      | Read full analysis   | Read the `creator-view.md` for the selected source's slug directory |
| 2      | Narrow search        | Prompt for additional filters (tag, type, source)                   |
| 3      | Show related         | Re-query with `--source=<selected-source>`                          |
| 4      | Show all from source | Re-query with `--source=<slug> --limit=50`                          |
| 5      | Done                 | Return to conversation                                              |

**Selection mechanism:** Present numbered options. User selects by number or
description. If user selects "Read full analysis," ask which source (by number
from results) if multiple sources are represented.

### 3.2 After Source Results

| Option | Label              | Action                                           |
| ------ | ------------------ | ------------------------------------------------ |
| 1      | View extractions   | Re-query with `--source=<selected-slug>`         |
| 2      | Read full analysis | Read the `creator-view.md` for the selected slug |
| 3      | Re-analyze         | Invoke `/analyze <source-url>`                   |
| 4      | Done               | Return to conversation                           |

### 3.3 After Stats Results

| Option | Label          | Action                                             |
| ------ | -------------- | -------------------------------------------------- |
| 1      | Browse by type | Re-query with `--target=sources --type=<selected>` |
| 2      | Browse by tag  | Re-query with `--tag=<selected-tag>`               |
| 3      | Search         | Prompt for free-text search terms                  |
| 4      | Done           | Return to conversation                             |

### 3.4 After Empty Results

| Option | Label          | Action                                |
| ------ | -------------- | ------------------------------------- |
| 1      | Broaden search | Remove most restrictive filter, retry |
| 2      | Show stats     | Run `--stats` to show available data  |
| 3      | List sources   | Run `--target=sources`                |
| 4      | Done           | Return to conversation                |

## 4. Auto-Rebuild Procedure

### 4.1 When to Trigger

The auto-rebuild runs when the SQLite database file does not exist at
`.research/content-analysis.db`.

**Detection:** Before running any query, the skill checks for the database file.
The `recall.js` script also checks and exits with an error message if missing.

### 4.2 Rebuild Steps

1. **Detect missing index.** Announce: "Index not found. Building from
   .research/ files..."

2. **Run rebuild.** Execute:

   ```bash
   node scripts/cas/rebuild-index.js
   ```

3. **Parse output.** The script reports:

   ```
   Results:
     Sources: N
     Extractions: M
     Unique tags: K
     Integrity: ok
     FK violations: 0
   ```

4. **Report to user.** "Done. N sources, M extractions indexed."

5. **Run the original query.** After rebuild completes, re-run the user's
   original `/recall` command.

### 4.3 Rebuild Failure Handling

If `rebuild-index.js` fails:

1. Report the sanitized error to the user.
2. Suggest manual inspection: "Check that .research/ contains analysis.json
   files. Run `node scripts/cas/rebuild-index.js --verbose` for details."
3. Do NOT silently fall back to file-based search. The index is the query
   interface.

### 4.4 Rebuild Flags

| Flag        | Purpose                                             |
| ----------- | --------------------------------------------------- |
| `--verbose` | Print per-source log lines during rebuild           |
| `--dry-run` | Report what would be indexed without making changes |

### 4.5 Data Sources for Rebuild

The rebuild script scans three directories for `analysis.json` files:

| Directory                            | Source type                         |
| ------------------------------------ | ----------------------------------- |
| `.research/analysis/<slug>/`         | Any (reads `source_type` from file) |
| `.research/repo-analysis/<slug>/`    | Typically `repo`                    |
| `.research/website-analysis/<slug>/` | Typically `website`                 |

It also reads `.research/extraction-journal.jsonl` for all extraction entries.

Directories starting with `_` are skipped (convention for archived or temporary
analysis runs).

## 5. Examples

### 5.1 Simple Free-Text Search

```
/recall architecture
```

Searches all extraction candidates and notes for "architecture" (stemmed: also
matches "architectural"). Returns up to 20 results ordered by insertion recency.

### 5.2 Free-Text with Type Filter

```
/recall plugin system --type=repo
```

Searches for "plugin system" only in extractions from repo sources. Useful when
you remember the concept came from a GitHub repo but not which one.

### 5.3 Tag-Based Filter

```
/recall --tag=anti-pattern
```

Returns all extractions tagged with "anti-pattern". No free-text search; purely
tag-based. Shows patterns to avoid that were identified across all analyzed
sources.

### 5.4 Combined Tag and Free-Text

```
/recall routing --tag=architecture --type=repo
```

Finds extractions matching "routing" AND tagged "architecture" AND from repo
sources. Three-way AND filter for precise results.

### 5.5 Sort by Novelty

```
/recall --type=repo --sort=novelty
```

Returns all repo extractions sorted by novelty (high first). No free-text
filter. Good for browsing the most novel ideas discovered across all analyzed
repos.

### 5.6 Source-Specific Query

```
/recall --source=unstructured
```

Returns all extractions from sources containing "unstructured" in the source
field. Shows everything extracted from a specific repo or website.

### 5.7 List All Sources

```
/recall --target=sources
```

Lists all analyzed sources grouped by type. Shows how many repos, websites,
documents, and media sources have been analyzed.

### 5.8 Sources Filtered by Type

```
/recall --target=sources --type=website
```

Lists only website sources. Shows title, quality band, fit classification, and
summary for each.

### 5.9 Index Statistics

```
/recall --stats
```

Shows total counts (sources, extractions, tags), breakdowns by source type and
novelty, and the top 15 most-used tags. Good for orientation before searching.

### 5.10 High-Novelty Patterns from Recent Analysis

```
/recall --sort=novelty --tag=pattern --limit=10
```

Returns the top 10 highest-novelty extraction candidates tagged as "pattern".
Useful for finding the most interesting transferable patterns across all
analyzed sources.

### 5.11 Broad Search with Increased Limit

```
/recall extraction --limit=50
```

Searches for "extraction" across all sources with a higher result limit. Default
is 20; increase when you want a comprehensive view.

### 5.12 Finding Knowledge Entries

```
/recall --tag=knowledge --type=website --sort=recent
```

Returns knowledge-type extractions from website sources, newest first. Good for
finding factual information and reference material from analyzed documentation
sites.

### 5.13 Cross-Source Pattern Discovery

```
/recall "file type detection" --sort=novelty
```

Quoted phrase search for an exact concept across all sources. The novelty sort
surfaces sources that contribute genuinely new approaches to the concept.

### 5.14 Checking What Needs Synthesis

```
/recall --target=sources --type=repo
```

Lists all repo sources. Check the `last_synthesized_at` column to see which
sources have been included in synthesis and which are pending. Sources with null
or stale timestamps need re-synthesis via `/analyze --synthesize --type=repo`.
