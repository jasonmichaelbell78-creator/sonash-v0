## üéØ Objectives & Requirements

### Primary Objectives

1. **Centralized Support Access**: Provide a single location for all
   support-related features
   - **Success Metric**: Users can access sponsor contact and help resources
     within 2 taps
   - **Priority**: P0

2. **Sponsor Contact Management**: Enable users to store and quickly reach their
   sponsor
   - **Success Metric**: Users can add sponsor info and call with 1 tap from
     Support Tab
   - **Priority**: P0

3. **Help Resource Availability**: Provide quick access to recovery resources
   and FAQs
   - **Success Metric**: Common questions answerable without leaving app
   - **Priority**: P1

### User Stories

**As a user in recovery, I want to quickly call my sponsor when I'm struggling
so that I can get immediate support.**

**Acceptance Criteria**:

- [ ] Can add sponsor name, phone number, and optional email
- [ ] Can call sponsor with single tap from Support Tab
- [ ] Can text sponsor with single tap
- [ ] Contact persists across sessions (Firestore)

**As a new user, I want to find help resources so that I can understand how to
use the app and find recovery support.**

**Acceptance Criteria**:

- [ ] FAQ section covers common questions
- [ ] Links to external recovery resources (AA, NA websites)
- [ ] Getting started guidance available

### Requirements

#### Functional Requirements

1. **Sponsor Contact Storage**: Users can save sponsor contact information
   - **Priority**: MUST
   - **Rationale**: Core feature - users need quick access to sponsor

2. **Quick-Dial Functionality**: One-tap calling/texting from stored contact
   - **Priority**: MUST
   - **Rationale**: Speed is critical during moments of need

3. **Help Resources Section**: Static content with FAQs and links
   - **Priority**: SHOULD
   - **Rationale**: Reduces user confusion, self-service support

4. **Multiple Contact Support**: Store more than one support contact
   - **Priority**: COULD
   - **Rationale**: Some users have multiple support people

#### Non-Functional Requirements

1. **Performance**: Support Tab loads in < 500ms
2. **Security**: Contacts stored in user's Firestore document (private)
3. **Accessibility**: All UI elements accessible, proper labels for screen
   readers
4. **Compatibility**: Works on iOS Safari, Chrome Android, desktop browsers

### Technical Approach

**Technology Stack**:

- Next.js App Router: `/app/support/page.tsx`
- shadcn/ui: Card, Button, Dialog, Accordion components
- Firestore: User subcollection for contacts
- React Hook Form: Contact add/edit form

**Key Components**:

1. **SupportTab** (`/app/support/page.tsx`): Main page component
2. **SponsorContactCard**: Display card with quick actions
3. **AddContactDialog**: Modal form for adding/editing contacts
4. **HelpResourcesSection**: Static content with FAQ accordion
5. **QuickDialButton**: Initiates tel: or sms: link

**Data Model**:

```typescript
// Firestore: users/{userId}/supportContacts/{contactId}
interface SupportContact {
  id: string;
  name: string;
  phone: string;
  email?: string;
  relationship: "sponsor" | "sponsee" | "friend" | "counselor" | "other";
  isPrimary: boolean;
  lastContacted?: Timestamp;
  createdAt: Timestamp;
  updatedAt: Timestamp;
}

// Quick Actions integration
interface QuickAction {
  id: string;
  label: string;
  icon: string;
  action: () => void;
}
```

### Design Decisions

**Decision 1: Subcollection vs Embedded Array**

- **Options Considered**: Embed contacts in user doc vs separate subcollection
- **Chosen Approach**: Subcollection (`users/{uid}/supportContacts`)
- **Rationale**: Allows unlimited contacts, individual security rules, easier
  querying
- **Trade-offs**: Extra Firestore reads, slightly more complex queries

**Decision 2: Static Help Content vs CMS**

- **Options Considered**: Hardcoded content, Firestore CMS, external CMS
- **Chosen Approach**: Hardcoded static content initially
- **Rationale**: Simpler to implement, content changes rarely, no admin UI
  needed yet
- **Trade-offs**: Requires code deploy for content updates
