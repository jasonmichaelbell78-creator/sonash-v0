## 1. Executive Summary

The dev dashboard is a developer-facing command center for the SoNash v0 project
— a dark-themed React/Next.js panel at `/dev` that makes 163+ persistent files
generated by 50+ skills visible in one place. The research confirms the project
already has a rich, well-structured data landscape suitable for a dashboard
without any new data collection infrastructure. The challenge is not generating
data — it is surfacing data that already exists.

The final tab structure is **six domain-organized tabs** plus a dashboard header
widget: Health & Alerts, Debt Pipeline, Code Review Quality, Build Pipeline &
Process Compliance, Governance & Audits, and Planning & Research. This structure
emerged from rigorous evaluation of three competing grouping models
(domain-based, temporal, action-oriented, and co-occurrence) and was settled by
user decision in Session #245. The 6-tab model covers all 36 HIGH-relevance data
sources identified in the full data inventory.

The **architecture is hybrid fetch**: in dev mode
(`process.env.NODE_ENV === 'development'`), all tabs fetch live data from
Next.js API routes (`/api/dashboard/<tab>`) that read JSONL/JSON files directly
from disk. In production (`npm run build` + Firebase Hosting static export), API
routes are absent and tabs fall back to pre-built JSON files in `public/`. A
`prebuild` npm hook runs `scripts/build-dashboard-data.js` to generate these
static snapshots. The existing `output: "export"` constraint in
`next.config.mjs` makes this split mandatory — API routes are dev-mode only.

**Critical data quality issues** were discovered during deep-dive analysis:
`velocity-log.jsonl` is confirmed broken (50 records, all `items_completed: 0`);
`commit-log.jsonl` is entirely seeded historical data (634 records, all
`seeded: true`, `branch: "seeded"`, `filesChanged: 0`); the archive portion of
`reviews-archive.jsonl` has all count fields set to 0 (useful only for pattern
frequency analysis, not numeric metrics). Three pre-work bugs in existing
scripts (BUG-01 through BUG-03) must be fixed before the dashboard build script
produces correct output. Four packages are not yet installed: `recharts`,
`@tanstack/react-table`, `@tanstack/react-virtual`, and `minisearch`.

The most actionable pre-work gate before any tab development starts is: (1)
install the four missing packages, (2) add five missing shadcn components, (3)
fix BUG-01 (lowercase status strings in `debt-health.js`), (4) verify G35
(confirm `override-log.jsonl` canonical path — confirmed as
`.claude/override-log.jsonl`), and (5) acknowledge the three BLOCKS-severity
data gaps (velocity broken, commit-log seeded, health-ecosystem-audit never
run). None of these gaps prevent building the dashboard — they require explicit
"Data Unavailable" widget states in specific places.

The two health log systems (`ecosystem-health-log.jsonl` showing D/67 and
`health-score-log.jsonl` showing B/87) measure genuinely different things and
should both be displayed. The D/67 score reflects technical health (code
quality, debt, tests); the B/87 reflects process health (session management,
hook compliance, review workflow). Showing only one would misrepresent the
project's health state. Similarly, the two warning systems
(`data/ecosystem-v2/warnings.jsonl` with lifecycle state machine and
`.claude/state/hook-warnings-log.jsonl` with raw hook events) must be unified
through a normalization layer rather than a join — their schemas are
incompatible and each captures different information.

## 3. Architecture

### 3.1 Overall Architecture

```
Developer opens /dev in browser
           |
    [Next.js Route: app/dev/page.tsx]
           |
    [Auth Gate: Firebase Admin claim required]
           |
    [dev-dashboard.tsx — DevTabProvider]
           |
    ┌──────────────────────────────────────────────┐
    │  Tab Navigation (6 tabs + header widget)     │
    │  DevTabId: health|debt|reviews|pipeline|     │
    │            governance|planning               │
    └──────────────────────────────────────────────┘
           |
    [Each tab fetches its own data]
           |
    ┌─────────────────┬─────────────────┐
    │ DEV MODE        │ PRODUCTION      │
    │ /api/dashboard/ │ /public/        │
    │ <tab> (live)    │ <tab>-data.json │
    │ reads JSONL     │ (pre-built)     │
    │ at request time │ static snapshot │
    └─────────────────┴─────────────────┘
           |
    [prebuild npm hook]
    [scripts/build-dashboard-data.js]
    [reads JSONL → field-strips → writes public/]
```

- Next.js inlines `process.env.NODE_ENV` at build time, so the dev branch is
  tree-shaken in production bundles.
- API routes directory (`app/api/`) does not exist and must be created from
  scratch.

- 87 raw events collapse to ~15 distinct warning types when deduplicated by

  (14), `propagation-staged` (10)

**Warning Unification Strategy [W3-T1A]:** The two warning systems cannot be
joined on a shared key (incompatible schemas). Use a normalization layer:

```typescript
interface UnifiedWarning {
  displayId: string; // lifecycle: id field; hook: "{hook}-{type}-{epoch}"
  source: "lifecycle" | "hook";
  timestamp: string; // lifecycle: pad date to midnight UTC; hook: use timestamp
  severity: "error" | "warning" | "info";
  message: string;
  category: string; // lifecycle: category field; hook: type field
  lifecycle?: "new" | "resolved";
  resolvedDate?: string;
  hook?: string;
  occurrences?: number;
  occurrencesSinceAck?: number;
  user?: string;
}
```

Sort: severity (error first), then timestamp (newest first). Deduplicate hook
warnings by `{hook}-{type}`, keep most recent, use `occurrences_since_ack` as
badge count.
