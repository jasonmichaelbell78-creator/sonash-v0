## Key Findings

---

### 1. MASTER_DEBT.jsonl Is the Single Source of Truth [CONFIDENCE: HIGH]

Every consumer ultimately traces back to `docs/technical-debt/MASTER_DEBT.jsonl`.
No consumer reads the raw intake pipeline files (deduped.jsonl, etc.) directly —
they all read either MASTER_DEBT.jsonl itself or derived artifacts
(metrics.json, views/*.md) that were generated from it.

The artifact chain is:

```
MASTER_DEBT.jsonl
  -> generate-metrics.js -> metrics.json + METRICS.md + logs/metrics-log.jsonl
  -> generate-views.js   -> INDEX.md + views/by-severity.md + views/by-category.md
                            + views/by-status.md + views/verification-queue.md
                            + LEGACY_ID_MAPPING.json
```

---

### 2. Views: Five Generated Files, One Stale Outlier [CONFIDENCE: HIGH]

`scripts/debt/generate-views.js` produces exactly five views plus INDEX.md.

| View file | Content | Status |
|-----------|---------|--------|
| `views/by-severity.md` | All items grouped S0-S3, with counts | Auto-generated, current (2026-03-26) |
| `views/by-category.md` | All items grouped by category (9 categories), sorted severity-first within each | Auto-generated, current (2026-03-26) |
| `views/by-status.md` | Items grouped NEW / VERIFIED / IN_PROGRESS / RESOLVED / FALSE_POSITIVE | Auto-generated, current (2026-03-26) |
| `views/verification-queue.md` | Items with status=NEW only; 2126 items currently | Auto-generated, current (2026-03-26) |
| `views/unplaced-items.md` | Items not assigned to any roadmap track | STALE — last updated 2026-02-01, says "Total Unplaced: 0" but is not regenerated by generate-views.js |
| `INDEX.md` | Summary totals by severity/category/status, all S0 items listed, first 20 S1 items | Auto-generated, current (2026-03-26) |

**Gap identified:** `unplaced-items.md` is NOT regenerated by generate-views.js.
It is a static document last written on 2026-02-01, claiming 0 unplaced items.
Given 8,472 total items as of today and a total claimed at the time of 868,
this file is approximately 4,600 items out of date. It will never update unless
someone explicitly calls the roadmap assignment script that originally wrote it.

**Missing view:** There is no view by `roadmap_ref` or `milestone`. Items have
these fields (preserved by generate-views.js in PRESERVED_FIELDS), but no view
groups them by roadmap track. The only roadmap-debt linkage lives in ROADMAP.md
itself.

### 3. Metrics: Point-in-Time Snapshot with Partial Historical Log [CONFIDENCE: HIGH]

`scripts/debt/generate-metrics.js` reads MASTER_DEBT.jsonl and writes:

- `docs/technical-debt/metrics.json` — machine-readable snapshot
- `docs/technical-debt/METRICS.md` — human-readable dashboard
- `docs/technical-debt/logs/metrics-log.jsonl` — append-only log (one line per generation)

**What metrics.json contains:**

| Field | Description |
|-------|-------------|
| `summary.total` | 8470 items (at last generation 2026-03-26) |
| `summary.open` | 7281 (total minus resolved and false positives) |
| `summary.resolved` | 1115 |
| `summary.resolution_rate_pct` | 13% |
| `by_status` | Counts for NEW/VERIFIED/IN_PROGRESS/RESOLVED/FALSE_POSITIVE |
| `by_severity` | Counts for S0/S1/S2/S3 |
| `by_category` | Counts per category (9 categories) |
| `by_source` | Counts per intake source (19 sources) |
| `alerts.s0_count` | 11 (non-RESOLVED/non-FP S0 items) |
| `alerts.s1_count` | 1259 (non-RESOLVED/non-FP S1 items) |
| `alerts.s0_items` | First 10 S0 items with id/title/file/line |
| `alerts.s1_items` | First 10 S1 items with id/title/file/line |
| `health.avg_age_days` | 41 |
| `health.oldest_age_days` | 55 |
| `health.verification_queue_size` | 2125 |

**Trend data:** metrics-log.jsonl has 112 entries, each with timestamp, total,
open, resolved, s0_alerts, s1_alerts. This is sufficient for trend computation
(the run-alerts.js `computeTrend` function uses it). However the schema is
limited: no per-category counts, no by_source, no avg_age. Trend granularity is
one entry per session-end run.

**Gap:** metrics.json includes only the first 10 S0 and S1 items in its alert
arrays. The full list requires reading MASTER_DEBT.jsonl directly. The actual
S0 count in metrics.json (11) does not match INDEX.md (26 S0 items) — this
discrepancy appears to be because metrics.json counts only non-resolved S0
items, while INDEX.md counts all items with severity=S0 regardless of status.
This is a documented distinction (open vs total) but could cause confusion.

### 4. Health Checker: debt-health.js Feeds Composite Score at 12% Weight [CONFIDENCE: HIGH]

`scripts/health/checkers/debt-health.js` reads metrics.json and MASTER_DEBT.jsonl
directly (does not go through the views). It produces 8 metrics:

| Metric | Benchmark (good/avg/poor) | Current value | Rating |
|--------|---------------------------|---------------|--------|
| s0_count | 0 / 0 / 1 | 11 | poor |
| s1_count | 0 / 5 / 10 | 1259 | poor |
| total_open | 10 / 30 / 60 | 7281 | poor |
| avg_age_days | 30 / 90 / 180 | 41 | average |
| resolution_rate | 50 / 30 / 10 (higher-better) | 13% | poor |
| intake_30d | 5 / 15 / 30 | 44 | poor |
| resolved_30d | 10 / 5 / 0 (higher-better) | 6 | average |
| net_flow | -5 / 0 / 10 | 38 | poor |

The `debt-health` checker maps to the **"Technical Debt" category** in the
composite health score, weighted at **12%** of the total composite score.
The Technical Debt category currently scores **21/100 (F-grade)**.

The composite health score (from ecosystem-health-log.jsonl) is currently
**63/100 (D-grade)**, down from 65 on 2026-03-25. Technical Debt is the
second-worst category after "Data Effectiveness."

**Bug note:** debt-health.js filters open items with
`d.status !== "resolved" && d.status !== "closed"` — it uses lowercase status
values, but MASTER_DEBT.jsonl uses uppercase ("RESOLVED", "FALSE_POSITIVE"). This
means it effectively does NOT filter resolved items for the avg_age calculation,
reading all 8470 items as "open" for that metric. This is likely causing
artificially deflated avg_age. The metrics.json calculation in generate-metrics.js
correctly uses `item.status !== "RESOLVED" && item.status !== "FALSE_POSITIVE"`.

### 5. /alerts Skill: Three Distinct Debt Checkers [CONFIDENCE: HIGH]

`run-alerts.js` (the `/alerts` script) has three separate debt-related checker
functions, plus a pending-refinements check that feeds into debt creation:

**a. `checkDebtMetrics()` — limited mode (always runs)**

- Reads `docs/technical-debt/metrics.json` for summary counts
- Reads `docs/technical-debt/MASTER_DEBT.jsonl` directly for rich grouping data
- Reads `docs/technical-debt/logs/metrics-log.jsonl` for trend
- Raises ERROR if s0 > 0 (with per-category breakdown and item listing)
- Raises WARNING if s1 > 10 (threshold from BENCHMARKS)
- Raises WARNING if open count trend is "increasing"
- Raises INFO with debt summary (total/open/resolved/resRate)
- Adds context: category groups, effort groups, sparklines, trend direction

**b. `checkDebtIntake()` — full mode only**

- Reads `docs/technical-debt/logs/intake-log.jsonl`
- Raises ERROR if intake_30d > 30
- Raises WARNING if intake_30d > 15
- Also tracks S0 intake rate (items entering as S0 per day)

**c. `checkDebtResolution()` — full mode only**

- Reads `docs/technical-debt/logs/resolution-log.jsonl`
- Raises ERROR if resolved_30d = 0
- Raises INFO if resolved_30d < 5

**d. `checkPendingRefinements()` — always runs, not debt-specific but feeds it**

- Items surfaced 3+ times auto-escalate to "S1 DEBT candidate"
- Alerts prompt user to create DEBT via `/add-debt`

**Freshness:** run-alerts.js reads metrics.json (cached/point-in-time from last
session-end), not MASTER_DEBT.jsonl directly for counts. The MASTER_DEBT.jsonl
read is supplementary (for enriching S0 item details).

**Weight in /alerts health score:** `debt-metrics` carries weight 0.11 (11%)
in the limited-mode scoring. `debt-intake` and `debt-resolution` are 0.02 and
0.01 respectively in full mode.

### 6. Session Start: No Direct Debt Data, Indirect Only [CONFIDENCE: HIGH]

The `session-start.js` hook does NOT read any debt files directly. It:

1. Deletes `.claude/tmp-alerts.json` (ephemeral cleanup)
2. Writes `.claude/hooks/.session-state.json` with session begin timestamp

The `/session-begin` skill (SKILL.md Phase 4.4) reads
`docs/technical-debt/INDEX.md` and notes S0/S1 counts for the pre-flight
summary. This is a human-readable summary step, not a script.

**Session-begin summary template** explicitly includes:
```
Warnings: N acknowledged | Tech Debt: N S0, M S1
```

This reads from INDEX.md, which is the auto-generated view (last regenerated
by generate-views.js). INDEX.md shows slightly different totals than metrics.json
because INDEX.md counts all items regardless of status while metrics.json
computes open vs. resolved separately.

### 7. Session End: Runs generate-metrics.js Every Session [CONFIDENCE: HIGH]

The `/session-end` skill (Phase 3, step 7e) runs
`node scripts/debt/generate-metrics.js` as a mandatory step — "never skip."

This means:
- `metrics.json` is updated at every session-end
- `METRICS.md` is regenerated
- `metrics-log.jsonl` gets a new append entry

`generate-views.js` is NOT in the session-end pipeline directly. Views are
only regenerated when `scripts/debt/consolidate-all.js` is run (step 7d in
session-end), since consolidate-all.js calls generate-views.js as part of its
pipeline. So views and INDEX.md do get refreshed each session-end, but only
as a side effect of the consolidation step.

### 8. Go Statusline: No Debt Data [CONFIDENCE: HIGH]

The Go statusline binary (`tools/statusline/`) has 22 widgets. None display
debt information. The widgets are:

- A1: Model name, A3: Session duration, A4: Permission mode, A6: Active agent
- B1: Git branch, B2: Project directory, B3: Worktree
- C1: Context gauge, C5: Rate limit 5hr, C6: Rate limit 7d, C7: Rate limit reset, C8: Lines changed
- D1: Hook health (reads hook-runs.jsonl), D5: Unacked warnings (reads hook-warnings-log.jsonl)
- E1: Current task, F4: Clock, F6/F7: Weather, F15: Uptime
- H2: GitHub PR status, H3: CI/CD pipeline
- I4: Session count today

The D1 widget shows hook health (pass/fail), not debt health. The D5 widget
shows unacked warnings from hook-warnings-log.jsonl, not debt warnings.
**No widget reads metrics.json, MASTER_DEBT.jsonl, or any debt-related file.**

This is a confirmed gap: the statusline is unaware of S0 debt count, total
open items, or resolution rate.

### 9. INDEX.md: Human Entry Point [CONFIDENCE: HIGH]

`docs/technical-debt/INDEX.md` is auto-generated by generate-views.js. It
contains:

- Total item count (8472 as of 2026-03-26)
- Quick stats table (severity, category, status counts)
- Links to all 5 views (note: includes link to unplaced-items.md which is stale)
- All S0 items listed in full (26 items)
- First 20 S1 items with a "...and N more" truncation
- Footer pointing to MASTER_DEBT.jsonl as canonical source

This is the file read by /session-begin for its debt snapshot.
The slight discrepancy between INDEX.md totals (8472) and metrics.json totals
(8470) reflects the fact that they were generated at slightly different times
and/or counts all-statuses vs open-only differently.

### 10. ROADMAP.md Integration [CONFIDENCE: HIGH]

ROADMAP.md contains hundreds of `DEBT-XXXX` references inline, tracking which
debt items are assigned to which roadmap tracks.

**reconcile-roadmap.js** (`scripts/debt/reconcile-roadmap.js`): Replaces legacy
CANON-XXXX IDs in ROADMAP.md with DEBT-XXXX IDs using LEGACY_ID_MAPPING.json.
It is a one-time migration/maintenance tool. Default is dry-run; requires `--write`
to modify. It does NOT run automatically.

**sync-roadmap-refs.js** (`scripts/debt/sync-roadmap-refs.js`): Validates that
DEBT-XXXX IDs referenced in ROADMAP.md and ROADMAP_FUTURE.md actually exist in
MASTER_DEBT.jsonl. Reports orphaned references. Also requires manual invocation
or CI integration.

**session-begin** reads only the first 100 lines of ROADMAP.md (Active Sprint
section). DEBT-XXXX items in the active sprint are visible only if they appear
in those first 100 lines.

**Gap:** Neither reconcile-roadmap.js nor sync-roadmap-refs.js runs in the
session-end pipeline. ROADMAP.md DEBT-XXXX reference validity is not checked
automatically.

### 11. Ecosystem Health Skill: Reads Cached Metrics Only [CONFIDENCE: HIGH]

`/ecosystem-health` (SKILL.md) invokes
`.claude/skills/ecosystem-health/scripts/run-ecosystem-health.js`, which calls
`scripts/health/run-health-check.js`. This orchestrates all 11 checkers
including `debt-health.js`.

The skill displays the Technical Debt category score as part of the 9-category
composite dashboard. Results are persisted to
`data/ecosystem-v2/ecosystem-health-log.jsonl`.

The `debt-health` checker reads `docs/technical-debt/metrics.json` (cached from
last session-end) plus MASTER_DEBT.jsonl for age calculations. It does not
re-run generate-metrics.js first. This means if MASTER_DEBT has changed since
the last session-end, the health check will use stale metrics counts but live
MASTER_DEBT data for age computation — a subtle inconsistency.

## Contradictions

**metrics.json vs INDEX.md totals:** metrics.json reports 8470 total items;
INDEX.md reports 8472. This is because they were generated at different times
(metrics.json at 02:12:29, INDEX.md at a later consolidation run) and reflects
the MASTER_DEBT.jsonl growing between runs. Not a bug — just race between runs.

**metrics.json s0_count (11) vs by-severity.md S0 count (26):** metrics.json
alerts.s0_count counts only open (non-RESOLVED, non-FALSE_POSITIVE) S0 items.
by-severity.md shows all 26 items regardless of resolution status. This is
intentional but not clearly labeled in METRICS.md.

**debt-health.js avg_age filter bug:** The checker uses lowercase status strings
(`"resolved"`, `"closed"`) while MASTER_DEBT.jsonl stores uppercase
(`"RESOLVED"`, `"FALSE_POSITIVE"`). This means the age calculation includes all
8470+ items including resolved ones, overstating the "open debt age" metric.
The metrics.json generation uses correct uppercase comparisons.

## Gaps

1. **Statusline has zero debt visibility.** An S0 critical item being added
   creates no statusline indicator. This is the most prominent gap for a
   real-time feedback loop.

2. **unplaced-items.md is permanently stale.** It was last written on
   2026-02-01 with a claim of "0 unplaced" but is not regenerated by any
   automated pipeline step. With 8472 total items and roadmap tracks that may
   not cover all categories, this view is unreliable.

3. **No roadmap-track view.** Items have `roadmap_ref` and `milestone` fields
   (preserved by generate-views.js) but there is no generated view showing
   "how many items per roadmap track." The only way to see this is filtering
   MASTER_DEBT.jsonl manually or reading ROADMAP.md inline references.

4. **sync-roadmap-refs.js not automated.** ROADMAP.md DEBT-ID references are
   never validated automatically. Orphaned references (pointing to resolved or
   deleted items) accumulate silently.

5. **Trend log schema is minimal.** metrics-log.jsonl records only
   total/open/resolved/s0_alerts/s1_alerts per entry. No per-category trend,
   no per-source trend, no age-distribution trend.

6. **debt-health.js benchmarks are wildly miscalibrated.** The "good" benchmark
   for `total_open` is 10 items. The project has 7,281 open items. Every debt
   metric scores 0 or near-0. This means the Technical Debt health category
   contributes almost nothing useful to the composite score — it is permanently
   saturated at "F."

7. **intake-log.jsonl and resolution-log.jsonl sparsity.** Only 80 intake
   entries and 14 resolution entries exist. The resolution-log records individual
   `resolved`/`false_positive` actions, not bulk operations. If items are bulk-
   resolved through consolidate-all.js or other scripts, those operations may
   not write to resolution-log.jsonl, making the `resolved_30d` metric
   unreliable.

8. **Session-begin debt snapshot is from INDEX.md, not metrics.json.** INDEX.md
   does not include trend data or resolution rate. The session-begin preflight
   summary gives a point-in-time count but no directional signal.

9. **/alerts full mode required for intake/resolution checks.** These two
   checkers only run in `--full` mode (42 categories). In `--limited` mode
   (the default, and the mode used by session-begin), only debt-metrics runs.
   Intake rate and resolution rate are invisible in normal session flow.

## Serendipity

**The consolidate-all.js → generate-views.js chain:** session-end does NOT call
generate-views.js directly. It calls consolidate-all.js (step 7d), which calls
generate-views.js internally. If consolidate-all.js is skipped or fails, views
will be stale even though metrics.json was just regenerated (step 7e runs after
7d). This creates a possible state where metrics.json is fresh but INDEX.md is
old.

**metrics.json.alerts.s1_count vs actual S1 count:** alerts.s1_count in
metrics.json is 1259, which is the count of open (non-resolved, non-false-
positive) S1 items. But by-severity.md shows 1360 total S1 items. The 101-item
discrepancy represents resolved/false-positive S1 items. The METRICS.md display
of "S1 High Priority Alerts (1259)" could be misread as "there are 1259 new
problems" rather than "1259 previously-identified items remain open."

**The `pending-refinements` checker in run-alerts.js is an implicit debt
intake path.** Items that are surfaced 3+ times without action auto-escalate to
"S1 DEBT candidate" alerts, prompting the user to use `/add-debt`. This is an
undocumented feedback loop from /alerts into the TDMS intake pipeline.
