## Key Findings

### 1. Full Rebuild is the Right Default — 179ms Parse Time Makes Incremental Unnecessary [CONFIDENCE: HIGH]

The entire MASTER_DEBT.jsonl (8,472 lines, ~7.6 MB) parses in 179ms on the local
machine (measured directly). A full rebuild of tdms.db from scratch on every
"Refresh Data" click is practical — no incremental complexity needed. This
conclusion comes from the actual timing measurement, not estimation.

The content_hash field exists on all items and is used for dedup identity by
sync-deduped.js, but using it for incremental sync buys nothing meaningful when
a full rebuild costs under 200ms including SQLite INSERT overhead (expected
under 2 seconds total for 8,472 rows in a transaction).

Incremental sync adds: a "last synced" cursor, per-row content_hash comparison,
INSERT-or-UPDATE logic (upsert), deletion detection for items removed from
MASTER, and test complexity. None of that complexity is justified for a
sub-second full rebuild.

**Recommendation:** Full rebuild only. Use `DELETE FROM items` + batch INSERT
inside a single SQLite transaction.

---

### 2. Transaction Wrapping is Non-Negotiable for Atomic Swap [CONFIDENCE: HIGH]

The existing codebase uses advisory file locking (safe-fs.js `withLock()`) and
atomic tmp-rename patterns (atomicWriteViaTmp) for all MASTER_DEBT writes. The
SQLite sync must follow the same atomicity discipline.

The correct pattern:

```
BEGIN TRANSACTION
  DELETE FROM debt_items;        -- clear current data
  INSERT INTO debt_items ...;    -- bulk insert all 8,472 rows
  DELETE FROM metrics_snapshots  -- optional: keep only last N
    WHERE id NOT IN (SELECT id FROM metrics_snapshots ORDER BY created_at DESC LIMIT 90);
  INSERT INTO metrics_snapshots ...;  -- append new snapshot row
COMMIT
```

If any step fails, ROLLBACK leaves the database in its prior valid state. The
web dashboard never reads a half-populated table. This is the SQLite equivalent
of the atomicWriteViaTmp pattern already in use.

The `better-sqlite3` npm package executes this natively synchronous with
`db.transaction(fn)()` — no async complexity, no connection pool, no ORM needed.

### 3. MASTER_DEBT Schema — All Fields Needed for the Database [CONFIDENCE: HIGH]

Sampled from actual MASTER_DEBT.jsonl line 1 (DEBT-0001). The canonical item
schema is:

```
source_id, source_file, category, severity, type, file, line,
title, description, recommendation, effort, status, roadmap_ref,
created, verified_by, resolution, content_hash, original_id,
merged_from (array), id (DEBT-XXXX), source, fingerprint (some items)
```

The `merged_from` field is an array — needs either a junction table or JSON
serialization in SQLite. Serializing as JSON text is simpler and acceptable
since the web dashboard does not need to JOIN on merged_from.

The `metrics_snapshots` table should match the structure of
`docs/technical-debt/metrics.json` exactly — generated by generate-metrics.js.
This file already exists and is the metrics contract.

### 4. generate-metrics.js is the Direct Analog — Sync Script Should Call It [CONFIDENCE: HIGH]

generate-metrics.js reads MASTER_DEBT.jsonl, computes the full metrics object
(summary, by_status, by_severity, by_category, by_source, alerts, health), then
writes metrics.json and METRICS.md. It also appends a row to
`docs/technical-debt/logs/metrics-log.jsonl`.

The sync script does NOT need to reimplement this logic. The correct approach:

1. Sync script reads MASTER_DEBT.jsonl.
2. Calls the same `calculateMetrics(items)` logic (either by requiring the
   module if refactored to export it, or by inlining the same computation).
3. Inserts the metrics snapshot row into `metrics_snapshots` table inside the
   same transaction as the debt_items rebuild.

**Critical observation:** generate-metrics.js is not currently exportable — it
uses a `main()` pattern with `process.exit()`. The sync script should inline the
metrics calculation rather than child-process-exec generate-metrics.js, to keep
the sync atomic and avoid the exit-code complication.

### 5. generate-views.js is NOT the Sync Script's Predecessor — It Writes Markdown [CONFIDENCE: HIGH]

generate-views.js generates markdown files (INDEX.md, by-severity.md,
by-category.md, by-status.md, verification-queue.md) and LEGACY_ID_MAPPING.json.
In `--ingest` mode (called by consolidate-all.js), it also appends new items
from raw/deduped.jsonl to MASTER_DEBT.jsonl.

The web dashboard replaces the markdown views, not generate-views.js itself.
generate-views.js continues to serve git-committed markdown for non-web
consumers (human review, CI checks). The sync script is additive, not a
replacement.

**Implication:** The sync script writes to data/tdms.db only. It does not
replace or disable generate-views.js.

### 6. consolidate-all.js Defines the Canonical Write-Path — Sync is Downstream [CONFIDENCE: HIGH]

consolidate-all.js runs 6 steps sequentially:

1. extract-sonarcloud.js (deprecated)
2. extract-audits.js
3. extract-reviews.js
4. normalize-all.js
5. dedup-multi-pass.js
6. generate-views.js --ingest (writes to MASTER_DEBT.jsonl)

The sync script belongs AFTER step 6 as an optional step 7. consolidate-all.js
should be extended to call sync-to-sqlite.js as a non-required final step (same
pattern as extract-sonarcloud.js: `required: false`). This makes the post-
consolidation DB fresh automatically without breaking existing flows.

### 7. sync-deduped.js Reveals the content_hash Identity Contract [CONFIDENCE: HIGH]

sync-deduped.js propagates severity/status changes from MASTER back to
raw/deduped.jsonl using content_hash as the join key. This is the canonical
identity mechanism: content_hash → DEBT-XXXX. Items without a content_hash are
identified by source_id or fingerprint.

For the SQLite schema, content_hash should be an indexed column (not PRIMARY KEY
— the DEBT-XXXX id is the primary key). Queries like "show all S0 items" are by
status/severity; content_hash lookup is only needed for sync operations.

### 8. safe-fs.js Advisory Locking Does NOT Apply to SQLite Sync [CONFIDENCE: HIGH]

safe-fs.js implements advisory locking via `.lock` files for JSONL writes. These
guards prevent concurrent MASTER_DEBT writes from the 30 scripts.

The sync script reads MASTER_DEBT.jsonl (read-only) and writes data/tdms.db.
SQLite has its own WAL-mode locking. The sync script does NOT need to acquire
the MASTER_DEBT.lock — reading a JSONL file mid-line-append is safe because the
sync reads the whole file in one readFileSync call. The only risk is reading
during an active write, but for a manual "Refresh" button this is acceptable.

If deterministic consistency is required, the sync script can optionally acquire
a read-lock (safe-fs `acquireLock`/`releaseLock`) before reading — but this is
defensive coding for a scenario that requires concurrent CLI + web button
simultaneous operation, which is unlikely for a desktop-only dev tool.

**Recommendation:** No locking in v1. Add read-lock in v2 if concurrent use
becomes a problem.

### 9. Trigger Points — Three Tiers with Different Priorities [CONFIDENCE: HIGH]

**Tier 1 (Manual, highest priority):** "Refresh Data" button on web → Next.js
API route (`app/api/debt/sync/route.ts`) → spawns
`node scripts/debt/sync-to-sqlite.js` via `child_process.execFile`. Returns
`{ status, itemCount, duration, timestamp }`.

The API route must be local-only or admin-gated. Since dev/page.tsx already
gates on Firebase admin claim, the route is protected by that same mechanism.

**Tier 2 (Automatic, post-consolidation):** Add sync-to-sqlite.js as step 7 in
consolidate-all.js's STEPS array with `required: false`. This keeps tdms.db
fresh after any full consolidation run. Cost: ~2 seconds extra on a pipeline
that already takes 30-60 seconds.

**Tier 3 (Automatic, post-resolve):** resolve-item.js and resolve-bulk.js both
call `writeMasterDebtSync()` or direct file mutations. A PostToolUse hook or a
`--sync-db` flag on these scripts could trigger a lightweight DB update.
However, given the 179ms full rebuild time, a full re-sync is cheaper than an
incremental patch and avoids divergence. Deferring this to v2 is reasonable.

**Not recommended:** Hook-based PostToolUse on debt-runner skill completion.
This fires after every AI tool call during a skill run, creating hundreds of
unnecessary sync operations. The right hook point is after session-end
consolidation, not after individual writes.

### 13. Error Handling Strategy — Partial Sync is Unacceptable [CONFIDENCE: HIGH]

Generate-metrics.js uses `process.exit(1)` on JSONL read failure and
`process.exit(2)` on write failure. The sync script should follow a different
pattern since it is invoked by the web API route and must return structured
errors rather than exit codes.

The pattern:

1. Read MASTER_DEBT.jsonl — if fails, return `{ error: 'READ_FAILED', ... }`
2. Parse all lines — skip malformed lines (same pattern as generate-metrics.js),
   collect `parseErrors` count.
3. If parseErrors > 1% of total (threshold: ~85 lines), abort sync and return
   `{ error: 'TOO_MANY_PARSE_ERRORS', parseErrors, total }`.
4. Begin SQLite transaction — if transaction fails, ROLLBACK and return
   `{ error: 'DB_WRITE_FAILED', ... }`.
5. On success, return `{ status: 'ok', itemCount, parseErrors, duration }`.

The 1% threshold is borrowed from generate-metrics.js's non-blocking warning
approach, but with a hard abort for sync since a partially-populated database is
worse than a stale one.

### 14. SQLite Package — better-sqlite3 is the Right Choice, Currently Absent [CONFIDENCE: HIGH]

No SQLite package exists in package.json. `better-sqlite3` is the correct choice
for this use case:

- Synchronous API (matches the existing synchronous JSONL pipeline style)
- Native Node.js bindings (fast: 8,472 inserts in a transaction takes ~100ms)
- No ORM needed — raw SQL for this use case
- Well-maintained, widely used in Next.js + Node.js projects

`better-sqlite3` requires a native build step (`node-gyp`). On Windows this
requires Visual Studio Build Tools. This is a known friction point on this
platform. Alternative: `@sqlite.org/sqlite-wasm` avoids native builds but is
async and browser-first — wrong fit. The native package is correct; just
document the build-tools prereq.

`drizzle-orm` or Prisma would add schema migration complexity that is
unnecessary for a single-file SQLite read mirror. Raw better-sqlite3 with
hand-written schema is the right level of abstraction.

### 15. Proposed SQLite Schema for data/tdms.db [CONFIDENCE: MEDIUM]

Based on actual MASTER_DEBT.jsonl item schema and generate-metrics.js output:

```sql
-- Primary debt items table
CREATE TABLE IF NOT EXISTS debt_items (
  id TEXT PRIMARY KEY,           -- DEBT-XXXX
  title TEXT NOT NULL,
  description TEXT,
  recommendation TEXT,
  category TEXT,
  severity TEXT,                 -- S0, S1, S2, S3
  status TEXT,                   -- NEW, VERIFIED, IN_PROGRESS, RESOLVED, FALSE_POSITIVE
  effort TEXT,                   -- E1..E5
  file TEXT,
  line INTEGER,
  source TEXT,
  source_id TEXT,
  source_file TEXT,
  type TEXT,
  created TEXT,                  -- ISO date string
  verified_by TEXT,
  verified_date TEXT,
  resolution TEXT,
  resolution_date TEXT,
  roadmap_ref TEXT,
  milestone TEXT,
  roadmap_phase TEXT,
  content_hash TEXT,
  fingerprint TEXT,
  merged_from TEXT,              -- JSON array serialized as text
  tags TEXT,                     -- JSON array serialized as text
  synced_at TEXT NOT NULL        -- ISO timestamp of sync run
);
CREATE INDEX IF NOT EXISTS idx_severity ON debt_items(severity);
CREATE INDEX IF NOT EXISTS idx_status ON debt_items(status);
CREATE INDEX IF NOT EXISTS idx_category ON debt_items(category);
CREATE INDEX IF NOT EXISTS idx_source ON debt_items(source);
CREATE INDEX IF NOT EXISTS idx_content_hash ON debt_items(content_hash);

-- Metrics snapshots (append-only, trend data)
CREATE TABLE IF NOT EXISTS metrics_snapshots (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  created_at TEXT NOT NULL,      -- ISO timestamp
  total INTEGER,
  open INTEGER,
  resolved INTEGER,
  false_positives INTEGER,
  resolution_rate_pct INTEGER,
  s0_count INTEGER,
  s1_count INTEGER,
  s2_count INTEGER,
  s3_count INTEGER,
  avg_age_days INTEGER,
  oldest_age_days INTEGER,
  metrics_json TEXT              -- Full metrics.json blob for complete access
);

-- Resolution events log
CREATE TABLE IF NOT EXISTS resolution_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  debt_id TEXT,
  resolved_at TEXT,
  pr_number INTEGER,
  resolution_type TEXT,
  raw_json TEXT
);

-- Intake events log
CREATE TABLE IF NOT EXISTS intake_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ingested_at TEXT,
  item_count INTEGER,
  source TEXT,
  raw_json TEXT
);

-- Sync metadata
CREATE TABLE IF NOT EXISTS sync_meta (
  key TEXT PRIMARY KEY,
  value TEXT
);
```

The `sync_meta` table holds last_sync_at, last_sync_duration_ms,
last_sync_item_count, master_file_size_bytes — surfaced in the dashboard header.

## Contradictions

**Incremental vs. full rebuild trade-off:** The presence of content_hash on
every item suggests the original design anticipated incremental sync
(sync-deduped.js uses it this way). However, the 179ms measured parse time
eliminates the performance argument for incremental complexity. These two
signals pull in opposite directions; the timing measurement is conclusive.

**DEBT-7593 (SQLite migration) vs. this sub-question:** DEBT-7593 proposes
eventually migrating the write side to SQLite (eliminating JSONL entirely). This
sub-question designs a JSONL-to-SQLite read mirror — a different, smaller scope.
The mirror design is compatible with DEBT-7593: the read mirror can be
decommissioned or converted to the canonical store when DEBT-7593 is
implemented. These are not contradictory architectures but sequential ones.

## Gaps

**better-sqlite3 on Windows:** No validation of whether `node-gyp` / Visual
Studio Build Tools are present in this environment. This is a known Windows
friction point. The package.json has no SQLite dependency at all — confirming
zero prior SQLite work. Testing the install is a pre-implementation step.

**Next.js API route exec pattern:** The pattern of a Next.js API route spawning
`child_process.execFile(node, [scriptPath])` is sound in development but
requires care in production builds (path resolution, script availability in
build output). The dev dashboard is desktop-only and runs in development mode,
which reduces this risk — but the API route design needs to account for the
working directory.

**Metrics-log.jsonl sync strategy:** The 113-row metrics-log.jsonl could be
synced on every "Refresh" (full reload of all 113 rows into metrics_snapshots)
or only the new rows since last sync. Given 113 rows is trivial, full reload on
every sync is correct for v1.

**Schema versioning:** No migration strategy has been designed. For a read
mirror that is rebuilt from scratch on every sync, schema migrations are simple:
drop the database and recreate it. A `db_schema_version` entry in `sync_meta` is
sufficient to trigger a schema refresh if the script detects a version mismatch.

**The dev-tabs.tsx "overrides" tab already exists** but has no implementation.
The override-log.jsonl data in .claude/state/ maps directly to this tab — the
sync architecture could trivially include it as a fifth table (override_events)

**generate-metrics.js logs every run** to metrics-log.jsonl (113 rows, one per
session-end or manual run). This log is the historical trend data the dashboard
needs for "resolution rate over time" charts. The data already exists and is
structured correctly. No new logging instrumentation is needed — just sync this
file into metrics_snapshots.

**The consolidate-all.js STEPS array uses `required: false` for deprecated
scripts** — this is the exact extension point for adding sync-to-sqlite.js as
step 7 without breaking existing CI or manual runs.

**The advisory locking in safe-fs.js already handles the concurrent-write
problem** that would make incremental sync dangerous. Since the sync script only
reads (and reads atomically via a single readFileSync), it sidesteps the locking
concern entirely.
