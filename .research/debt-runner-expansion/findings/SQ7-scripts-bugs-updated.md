### New Node: SQLite Sync (Where It Fits)

```
                                 generate-metrics.js
                                              │
                                    ┌─────────┴──────────────────────┐
                                    ▼                                 ▼
                               metrics.json                    METRICS.md
                               (machine-readable)
                                    │
                                    ▼

                                    │                  for item-level detail
                                    ▼
                              web.db (SQLite)
                                    │
                          ┌─────────┴──────────┐
                          ▼                     ▼
                    Web API server        metrics tables
                    (read-only)           trend tables
                                          item tables
```

**Where sync-to-sqlite.js should be positioned:**

After `generate-metrics.js` and before any web API request is served.
Specifically:

1. **After `generate-metrics.js`** — metrics.json is the freshest aggregate data
   and is what the web dashboard cares about most.
2. **After `MASTER_DEBT.jsonl` mutations** — any mode that mutates MASTER_DEBT
   (verify, sync, cleanup) should have generate-metrics.js as a post-step,
   followed by the SQLite sync. The existing post-mutation pipeline
   (resolve-bulk → generate-views → generate-metrics) already places
   generate-metrics at the end — sync-to-sqlite.js should be appended to this
   chain.
3. **Reading both `metrics.json` AND `MASTER_DEBT.jsonl`** — metrics.json
   provides aggregate counts/alerts efficiently. MASTER_DEBT.jsonl provides
   item-level detail (by severity, category, status) for the item list views.
   metrics-log.jsonl provides historical trend data.

**Data files sync-to-sqlite.js needs to read:**

- `docs/technical-debt/metrics.json` — aggregate metrics (primary read)
- `docs/technical-debt/MASTER_DEBT.jsonl` — item-level data for list/drill-down
  views
- `docs/technical-debt/logs/metrics-log.jsonl` — historical trend snapshots
- `docs/technical-debt/logs/intake-log.jsonl` — intake 30d count (for trend
  data)
- `docs/technical-debt/logs/resolution-log.jsonl` — resolved 30d count

## Part 4: generate-metrics.js — metrics-log.jsonl Gap Confirmed [CONFIDENCE: HIGH]

**Question from SQ4b:** Does generate-metrics.js log `by_source` and
`by_category` breakdowns to `metrics-log.jsonl`?

**Answer: NO.** This gap is confirmed by reading the `logMetricsGeneration()`
function in generate-metrics.js (lines 335–356).

The log entry written to metrics-log.jsonl is:

```json
{
  "timestamp": "...",
  "total": N,
  "open": N,
  "resolved": N,
  "s0_alerts": N,
  "s1_alerts": N
}
```

The fields `by_source` and `by_category` (which are computed in
`calculateMetrics()` and available in the `metrics` object) are **not included**
in the log entry.

**Confirmed by cross-referencing actual log file:** A sample line from
`docs/technical-debt/logs/metrics-log.jsonl` reads:

```json
{
  "timestamp": "2026-02-01T21:49:51.637Z",
  "total": 868,
  "open": 868,
  "resolved": 0,
  "s0_alerts": 18,
  "s1_alerts": 139
}
```

No `by_source` or `by_category` fields present.

**Web dashboard impact:** The trend chart data stored in metrics-log.jsonl
cannot show "breakdown by category over time" or "breakdown by source over time"
because the historical snapshots don't include those fields. This is a **data
gap** that must be addressed before building a trend-over-time view for
category/source breakdowns. Options: (a) expand logMetricsGeneration() to
include these fields, (b) have sync-to-sqlite.js compute the trend data fresh
from MASTER_DEBT.jsonl each time (expensive for 8,470+ items), or (c) accept
limited trend views at launch (summary only, not breakdown).

### BUG-01: debt-health.js uses lowercase status strings [CONFIRMED PRESENT]

**Location:** `scripts/health/checkers/debt-health.js`, line 65

**Current code (verified by direct read):**

```js
const openDebt = allDebt.filter(
  (d) => d.status !== "resolved" && d.status !== "closed"
);
```

**Problem:** MASTER_DEBT.jsonl uses uppercase statuses (`"RESOLVED"`, no
`"closed"` status exists). This filter never matches any item, so `openDebt`
always equals `allDebt`. The `avg_age_days` metric is computed over ALL items
including resolved ones, producing an inflated value.

**Web dashboard impact: HIGH** — The health mode dashboard and any web dashboard
consuming debt-health.js output will show incorrect avg_age_days for open items.
The benchmark comparison (good: 30d, average: 90d, poor: 180d) will produce
wrong scores because resolved items (many of which are old) inflate the average
age.

**Fix before or after web dashboard?** BEFORE. If the web dashboard sources
avg_age data from debt-health.js or from metrics that debt-health.js populates,
the number will be wrong from day one. The fix is trivial: change `"resolved"`
to `"RESOLVED"` and remove `"closed"`.

**Priority change in hybrid architecture:** ELEVATED. Was a CLI-only display
bug; now affects web dashboard data integrity.

### BUG-02: dedup-multi-pass.js has no --dry-run or --force flags [CONFIRMED PRESENT]

**Location:** `scripts/debt/dedup-multi-pass.js`

**Verification:** Grep for `dry.run|--force|process\.argv` returns no matches.
The script has no argument parsing at all — it always runs all 6 passes and
writes all output files unconditionally.

**Problem:** `debt-runner/REFERENCE.md` documents `--dry-run` and `--force` as
valid flags for this script. When debt-runner's dedup mode calls
`dedup-multi-pass.js --dry-run` (Step 1), it actually runs a full live dedup
pass. Users believe they are previewing, but deduplication is applied
immediately.

**Web dashboard impact: LOW** — This is a CLI workflow correctness bug, not a
data quality bug. The dedup output (raw/deduped.jsonl) feeds into MASTER_DEBT
via generate-views --ingest, which is a separate step. The web dashboard reads
from metrics.json/MASTER_DEBT.jsonl, not from the dedup pipeline directly.

**Fix before or after web dashboard?** AFTER. This is an important correctness
issue for the CLI workflow but does not affect web dashboard data quality. It
should be in the backlog but is not a blocker.

**Priority change in hybrid architecture:** UNCHANGED. Remains a CLI-only issue.

### BUG-03: resolve-bulk.js does not call sync-deduped.js [CONFIRMED PRESENT]

**Location:** `scripts/debt/resolve-bulk.js` — no call to sync-deduped.js in any
code path (verified by full read of the file). Compare: resolve-item.js
explicitly calls sync-deduped.js in both applyFalsePositive() and
applyResolved().

**Problem:** After every CI-triggered bulk resolution (via resolve-debt.yml),
`raw/deduped.jsonl` drifts out of sync with MASTER_DEBT.jsonl. If
consolidate-all.js is then run, generate-views.js --ingest reads from the stale
deduped.jsonl and can overwrite the resolution status changes.

**Web dashboard impact: MEDIUM** — The SQLite sync reads from MASTER_DEBT.jsonl
and metrics.json (both updated by resolve-bulk.js). The web dashboard data would
be correct immediately after a bulk resolution. The risk is indirect: if someone
runs consolidate-all.js after a CI resolution without first running
sync-deduped.js, MASTER_DEBT gets overwritten with the pre-resolution state, and
the next SQLite sync would push stale (wrong) data to the web.

**Fix before or after web dashboard?** Before the web dashboard goes live (but
not urgently before development starts). The fix is adding
`sync-deduped.js --apply` to resolve-bulk.js's post-write chain. The risk is
data corruption in the web dashboard if the consolidate-all.js race condition is
triggered.

**Priority change in hybrid architecture:** ELEVATED. In the CLI-only world, the
consequence of drifted deduped.jsonl is eventual manual discovery. In the hybrid
world, the consequence is the web dashboard showing wrong data after a CI
resolution if anyone runs the pipeline without the manual sync step.

### BUG-04: TRIAGED status in resolve-bulk.js is not a valid schema status [CONFIRMED PRESENT]

**Location:** `scripts/debt/resolve-bulk.js`, line 34:

```js
const ELIGIBLE_STATUSES = ["VERIFIED", "IN_PROGRESS", "TRIAGED"];
```

`scripts/config/audit-schema.json` line 23:

```json
"validStatuses": ["NEW", "VERIFIED", "FALSE_POSITIVE", "IN_PROGRESS", "RESOLVED"]
```

**Problem:** TRIAGED is in the CI eligibility check but not in the canonical
schema. Any item set to TRIAGED status would pass CI resolution (get marked
RESOLVED) but would fail schema validation (validate-schema.js would error on
it). No script was found that sets status to TRIAGED, making this a latent bug:
it would only trigger if a user manually edits MASTER_DEBT.jsonl to set
status=TRIAGED.

**Web dashboard impact: LOW** — The web dashboard would display any TRIAGED item
with an unexpected status label (if status is displayed). Schema validation
would catch it. No items currently have TRIAGED status.

**Fix before or after web dashboard?** AFTER. Latent bug with no current
trigger. Fix is simple (remove TRIAGED from ELIGIBLE_STATUSES) but needs
confirmation that TRIAGED was never intentionally planned as a status.

**Priority change in hybrid architecture:** UNCHANGED. Still a latent bug.

### BUG-05: verify-resolutions.js NEW→VERIFIED promotion does not check issue existence [CONFIRMED PRESENT]

**Location:** `scripts/debt/verify-resolutions.js` (verified in prior research,
not re-read — carry forward HIGH confidence from SQ7-v1).

**Problem:** Step 3 promotes items from NEW to VERIFIED solely if the referenced
file exists AND the line count is valid. It does NOT check whether the actual
issue still exists in the code. Items referencing refactored files where the
issue was incidentally resolved still get promoted to VERIFIED rather than
RESOLVED.

**Web dashboard impact: MEDIUM** — The web dashboard would show these items as
VERIFIED (implying active issues to fix) when they may actually be resolved. The
verification queue size (currently 2,125 items) shown in the dashboard would be
inflated by false VERIFIED items. Trends showing "items fixed per week" would be
understated.

**Fix before or after web dashboard?** AFTER. The promotion algorithm's weakness
is a known limitation — it requires AI-assisted verification (which debt-runner
verify mode provides) to truly assess. The web dashboard should note that
VERIFIED counts are lower-confidence than RESOLVED counts.

**Priority change in hybrid architecture:** LOW. The web dashboard makes the
problem more visible (showing inflated VERIFIED counts) but doesn't create new
data integrity risk.

### BUG-06: metrics-log.jsonl does not log by_source or by_category [CONFIRMED PRESENT]

**Location:** `scripts/debt/generate-metrics.js`, `logMetricsGeneration()`
function (lines 335–356).

**Current logged fields:** timestamp, total, open, resolved, s0_alerts,
s1_alerts

**Missing from log:** by_source (object), by_category (object), by_severity
(object — though severity counts can be inferred from alerts), avg_age_days,
resolution_rate_pct

**Problem:** The metrics-log.jsonl cannot be used to build trend-over-time
charts for category or source breakdowns. The data exists in metrics.json
(current state only) but historical trend data for categories/sources is lost.

**Web dashboard impact: HIGH** — This is a direct blocker for the "trend by
category" and "trend by source" views that a web dashboard would logically
include. A basic bar chart of "current state" works fine from metrics.json. But
a "debt by category over time" trend chart requires historical data in
metrics-log.jsonl.

**Fix before or after web dashboard?** The log schema should be expanded BEFORE
the web dashboard is built, so that trend data starts accumulating from launch.
If expanded after, the trend charts would only show data from the expansion date
forward. The fix is adding these fields to the `logEntry` object in
`logMetricsGeneration()`.

**Priority change in hybrid architecture:** ELEVATED from MEDIUM to HIGH.
Previously a minor gap (logs were barely used). Now a web dashboard data
architecture decision that affects what trend views are possible.

## Part 7: New Scripts Needed for Hybrid Architecture

### 7a. sync-to-sqlite.js (Required — does not exist)

**Does not exist.** Glob confirms no `sync-to-sqlite*` file in `scripts/debt/`.

**Purpose:** Reads metrics.json and (optionally) MASTER_DEBT.jsonl,
writes/updates a SQLite database for web API consumption. Replaces or
supplements the current CLI-terminal health display.

**Proposed inputs:**

- `docs/technical-debt/metrics.json` — aggregate metrics snapshot
- `docs/technical-debt/MASTER_DEBT.jsonl` — item-level data (for drill-down
  views)
- `docs/technical-debt/logs/metrics-log.jsonl` — historical trend snapshots
- `docs/technical-debt/logs/intake-log.jsonl` — intake trend data
- `docs/technical-debt/logs/resolution-log.jsonl` — resolution trend data

**Proposed outputs:**

- `data/web.db` (or `docs/technical-debt/web.db`) — SQLite file

**Proposed tables:**

- `metrics_snapshots` — one row per metrics-log.jsonl entry (for trend charts)
- `debt_items` — one row per MASTER_DEBT.jsonl item (for list/search views)
- `intake_events` — intake-log.jsonl rows (for intake trend)
- `resolution_events` — resolution-log.jsonl rows (for resolution trend)

**When to run:** After `generate-metrics.js` completes. Natural trigger points:

- End of every debt-runner mode that mutates MASTER_DEBT (append to existing
  post-mutation chain: generate-metrics → sync-to-sqlite)
- Via `npm run tdms:sync-web` (new package.json script)
- CI pipeline after resolve-debt.yml runs resolve-bulk.js

**Note on naming:** The SQ1b research (SQLite sync architecture findings) may
have determined whether the file should be named `sync-to-sqlite.js` vs
`sync-to-web.js`. This agent defers to the SQ1b findings for the canonical name.

**Not a new script but a required modification.** The `logMetricsGeneration()`

### 7c. Web-specific data transformation scripts (Not required at launch)

No additional extraction or transformation scripts appear needed at launch. The
metrics.json schema is rich enough for a v1 web dashboard:

- Summary counts (total, open, resolved, false_positives, resolution_rate_pct)
- by_severity breakdown
- by_category breakdown
- by_source breakdown
- alerts (s0_items, s1_items with file/line)
- health (avg_age_days, oldest_item, verification_queue_size)

The only transformation needed is the SQLite sync itself (sync-to-sqlite.js).
More complex transformations (e.g., trend lines, velocity, cohort analysis) can
be computed by the web API from the raw SQLite tables rather than requiring
dedicated scripts.

## Serendipity

**resolve-debt.yml post-resolution gap:** The CI workflow resolve-debt.yml calls
resolve-bulk.js → generate-views.js but does NOT call generate-metrics.js. This
means after every CI resolution event, metrics.json is stale. In the CLI-only
world this is a minor issue (next session-end run fixes it). In the hybrid
world, the web dashboard would show stale counts until the next manual metrics
run. The sync-to-sqlite.js build plan should include adding
`generate-metrics.js` to the resolve-debt.yml workflow.

**debt-health.js is not called by any automated pipeline.** It is only called by
the debt-runner health mode and by the health system check runner. Its output
does not feed into metrics.json. A web dashboard built on metrics.json would
bypass debt-health.js entirely — meaning BUG-01's incorrect avg_age_days would
not affect the web dashboard if the dashboard reads avg_age_days directly from
metrics.json (which calculates it correctly using uppercase "RESOLVED"). The
BUG-01 priority assessment above assumed debt-health.js output would be surfaced
in the web dashboard; if the dashboard reads metrics.json directly, BUG-01 is
lower priority.
