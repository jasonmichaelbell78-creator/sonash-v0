  <action>
    Fix the `buildMechanisms` function in `scripts/reviews/build-enforcement-manifest.ts` to eliminate the inflated Semgrep matching.

    Current buggy code (around line 380-387):
    ```typescript
    for (const [ref, ruleId] of sources.semgrepRules) {
      const refSlug = slugify(ref);
      if (fuzzyMatch(refSlug, slug) || ref === pattern.category || ref === pattern.name) {
        mechanisms.semgrep = `active:${ruleId}`;
        break;
      }
    }
    ```

    Replace the Semgrep matching block with STRICT matching:
    ```typescript
    for (const [ref, ruleId] of sources.semgrepRules) {
      const refSlug = slugify(ref);
      // Exact slug match only -- no category-level or fuzzy matching
      if (refSlug === slug) {
        mechanisms.semgrep = `active:${ruleId}`;
        break;
      }
    }
    ```

    This removes:
    1. `ref === pattern.category` -- the primary bug causing category-wide false matches
    2. `ref === pattern.name` -- redundant since slugify(ref) === slugify(pattern.name) covers this
    3. `fuzzyMatch(refSlug, slug)` -- the 60% word overlap matcher that creates unrelated matches

    Replace with exact slug equality (`refSlug === slug`). This means a Semgrep rule with `code-pattern-ref: "Error Sanitization"` will ONLY match the pattern whose slugified name is "error-sanitization".

    Also remove or comment out the `fuzzyMatch` function entirely (lines 334-346) since it is no longer called by any code path. Check if fuzzyMatch is used by regex or ESLint matching first -- if so, leave it for those but remove it from Semgrep matching only.

    WAIT: Check if fuzzyMatch is still used for regex and ESLint matching (lines 365-378). If YES, keep the function but only remove it from the Semgrep block. If NO (regex and ESLint also should use exact matching), remove it entirely.

    Decision: For regex and ESLint rules, fuzzyMatch may be acceptable since those rule IDs are designed to slugify-match pattern names. But for safety and consistency, consider switching regex and ESLint to exact matching too. The safest approach: keep fuzzyMatch for regex/ESLint (their rule IDs were designed with fuzzy matching in mind) but use exact for Semgrep (where we now control the refs).
